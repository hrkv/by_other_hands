# План разработки: By Other Hands

## 1) Кратко о проекте
- **Жанр / формат**: онлайн пошаговая тактическая 2D-игра для двух игроков
- **Платформы**: Web (современные десктопные браузеры, по возможности мобильные)
- **Движок / стек**: 2D-движок под Web (кандидаты: Phaser, PixiJS + свой слой логики, Godot Web export; финальный выбор — на этапе A)
- **Целевая сессия**: 10–30 минут (одна партия или несколько быстрых)
- **Питчи (1–2 предложения)**: Два игрока начинают на своих клетках‑“домах” и ходят по клеточному полю в пошаговом режиме, но каждый делает ходы **за фишку соперника**, а не за свою. Цель — своей фишкой добраться до чужого дома, а так как напрямую ты управляешь только чужой фишкой, продвижение к цели достигается за счёт грамотного использования бустов на поле.

## 2) Цели и ограничения
- **Главная цель**: довести игру до состояния “играбельный релиз” с минимальным ядром и понятным дальнейшим расширением.
- **Ограничения**:
  - время/ресурсы команды: ориентир — 1–2 разработчика, ночные/вечерние слоты; планируемый горизонт до первого публичного релиза: 2–4 месяца.
  - обязательные фичи: онлайн‑матч на двоих, базовый матчмейкинг/подключение, один тип поля с “домами”, минимальный набор бустов, система ходов “за соперника”, базовый UI (поле, очередь хода, подсказка цели).
  - “не делаем” (scope guardrails): сложная прогрессия аккаунта, перманентные мета‑системы, кастомизация скинов, масштабный лор/нарратив, AI‑боты (минимум на первый релиз), более 1–2 типов арен.

## 3) Игровые основы (design pillars)
Заполнить 3–5 опорных принципов, которые помогают принимать решения:
- **Pillar 1**: “Ходишь за другого” — основная механика про контроль чужих ходов, а не своих.
- **Pillar 2**: “Чтение намерений” — игра про угадывание планов оппонента и создание для него “ловушек” через выгодные на первый взгляд ходы.
- **Pillar 3**: “Простые правила, сложные решения” — минимум механик, но высокий простор для планирования и обыгрывания соперника.
- **Pillar 4** (опционально): “Короткая, но насыщенная партия” — одна игра должна укладываться в 10–30 минут.

## 4) Вертикальный срез (Vertical Slice) — главный ранний результат
**Цель**: 5–15 минут цельного геймплея, демонстрирующего ядро.

### Состав вертикального среза
- **Core loop**: матчмейкинг/подключение → игроки начинают на своих клетках‑“домах” → по очереди выбирают ход **для фишки соперника** → срабатывание эффектов клетки/бустов, которые опосредованно продвигают к цели свою фишку или мешают фишке оппонента → обновление положения и состояния поля → проверка победы (чья фишка первой дошла до чужого дома) → следующий ход или конец партии.
- **Один уровень/сцена/арена**: фиксированное прямоугольное поле из клеток с одной базой для каждого игрока и набором простых бустов.
- **Базовый UI/UX**: экран лобби/подключения, отображение поля и фишек двух игроков, подсветка доступных ходов, индикация очереди хода, подсказка цели.
- **Аудио-заглушки**: базовые клики/перемещения фишек, простой фон/эмбиент.
- **Технические требования**: стабильное поведение в браузере, корректная передача ходов между двумя клиентами, защита от рассинхронизации, возможность перезагрузить страницу и вернуться в текущую партию (минимально).

### Критерии готовности вертикального среза
- **Играбельно без объяснений** (пользователь понимает цель за 30–60 сек)
- **Нет блокирующих багов** (crash/softlock)
- **Собирается и запускается одним шагом** (скрипт/команда)

## 5) MVP (Minimum Viable Product)
**Цель**: релиз, который можно продавать/раздавать и который “держится” без постоянных правок.

### Состав MVP
- **Игровое ядро**: полностью реализованный core loop
- **Контент минимум**: 3–5 “единиц” контента (уровни/миссии/враги/загадки — что применимо)
- **Прогресс/сессии**: сохранения/чекпоинты/перезапуск (что нужно)
- **Обучение**: интро/подсказки/первая миссия
- **Настройки**: управление, громкость, графика (минимум)
- **Доступность (минимум)**: переназначение клавиш или альтернативы, масштаб UI (по возможности)
- **Стабильность**: отсутствие крашей на целевой платформе

### Критерии “MVP готов”
- **Можно пройти/закончить** без софтлоков
- **Понятные цели** на каждом отрезке
- **Стабильная производительность** на целевом железе (зафиксировать метрику)

## 6) Этапы работ (milestones)
Ниже — рекомендуемая последовательность. Длительности даны ориентировочно для 1–2 человек при частичной занятости; при изменении стека/команды можно скорректировать.

### Этап A — База проекта (1–3 дня)
- Создать рабочую структуру проекта (папки, соглашения по именованию)
- Наладить сборку/запуск “одной командой”
- Минимальный pipeline ассетов (импорт, форматы, ограничения)
- Протокол версий: SemVer в упрощённом виде — `0.MAJOR.MINOR` до “большого” релиза (0.x.y), затем переход на `1.x.y`; патч‑версии инкрементируются только при исправлении багов без изменения правил.

**Артефакты**: `README.md` (как запустить), `PLAN.md` (этот файл), базовая структура папок.

### Этап B — Прототип ядра (3–7 дней)
- Реализовать core loop в “сером боксе” (без красивого арта)
- Минимальная обратная связь (звук/частицы/вибрация — заглушки)
- Инструменты отладки (HUD debug, консоль, hot-reload — по стеку)

**Критерий**: уже “интересно” в течение 2–3 минут.

### Этап C — Вертикальный срез (7–14 дней)
- Одна полноценная игровая “порция”: уровень/миссия/забег
- Базовый UI/UX (меню, пауза, финиш)
- Первичная балансировка

**Критерий**: можно дать другу и получить связный фидбек.

### Этап D — MVP контент + системы (10–20 дней)
- Довести контент до минимального объёма
- Сохранения/прогресс/мета (если есть)
- Устойчивость к ошибкам (корректные перезапуски, no softlocks)

### Этап E — Полировка + оптимизация (5–10 дней)
- Производительность, загрузки, размер билда
- UX-правки по результатам плейтестов
- Стабилизация, фиксы, “заморозка” фич

### Этап F — Релиз (2–5 дней)
- Подготовка страницы (описание, скриншоты, трейлер — по необходимости)
- Финальная сборка, теги/версии, заметки о релизе

## 7) Технический план (без привязки к движку)
### Архитектура
- **Данные**: предпочтительно data-driven (JSON/TOML/ресурсы движка) для баланса/контента
- **Игровые системы**: разнести “модель” и “представление” (логика отдельно от графики)
- **Состояния игры**: меню → игра → пауза → результат (явный state machine)

### Инструменты качества
- **Логи**: уровни (debug/info/warn/error), лог в файл на релизных сборках
- **Краш-репорты**: минимально — сохранение стека/логов; лучше — интеграция сервиса (опционально)
- **Проверки**: линтер/форматтер (если применимо), pre-commit (опционально)

## 8) Плейтесты и метрики
- **Плейтест 1 (после прототипа)**: понять “цепляет ли” core loop
- **Плейтест 2 (после вертикального среза)**: ясность целей, сложность, темп
- **Плейтест 3 (перед релизом)**: стабильность, UX, производительность

**Минимальные метрики** (выбрать релевантные):
- время до “понимания цели”
- % завершивших вертикальный срез
- места частых смертей/провалов/выходов

## 9) Риски и меры
- **Разрастание scope**: фиксировать “не делаем” и делать фичи только через критерий pillars
- **Контент дороже кода**: выбрать формат контента, который быстро штампуется
- **Техдолг прототипа**: запланировать “перепрошивку” после подтверждения fun-factor
- **Перформанс**: ранние профили, лимиты ассетов, бюджет на кадр

## 10) Бэклог (черновой шаблон)
Заполняется после выбора движка/жанра.

### Must (для MVP)
- [ ] Core loop (финальная версия)
- [ ] UI: пауза/настройки/результаты
- [ ] Сохранения/прогресс (если нужно)
- [ ] 3–5 единиц контента
- [ ] Туториал/онбординг
- [ ] Баланс базовый
- [ ] Стабилизация (0 crash, 0 softlock)

### Should (если остаётся время)
- [ ] Достижения/статистика
- [ ] Доп. режим/модификаторы
- [ ] Улучшенная доступность

### Could (после релиза)
- [ ] Доп. контент/уровни
- [ ] Моды/редактор

## 11) Определение готовности (Definition of Done)
- Фича покрыта тестом/проверкой (если применимо) или имеет понятный ручной тест-план
- Нет регрессий в основных сценариях (запуск → игра → пауза → выход)
- Не ломает производительность за пределами бюджета
- Обновлена документация/настройки, если нужно
